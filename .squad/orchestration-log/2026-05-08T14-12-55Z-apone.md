# Apone — Architecture Review: Container Topology Decision Gate

**Date:** 2026-05-08T14:12:55Z  
**Role:** Architect + Senior Technical Decision-Maker (Apone)  
**Task:** Investigate container app deployment architecture and recommend path forward

## Problem Summary

**User Question:** "Why does dev have a container app deployed for each agent but test only has 2 container apps?"

**Finding:** Mismatch between intended architecture and deployed reality.

---

## Architecture Analysis

### Current Reality (Verified)

**Setup-Solution.ps1 deploys identical topology to all environments:**
- 1× `.NET Tickets API` Container App
- 1× `Python Resolution API` Container App
- **Total: 2 containers per environment**

Both dev and test execute the same script; no environment branching.

### Dev Extra Containers (Not from Setup-Script)

- `ca-mcp` — MCP Server candidate (stub module: 0 bytes, disabled)
- `ca-incident`, `ca-classifier`, `ca-request`, `ca-escalation`, `ca-resolution` — Artifact of hosted-agents migration experiment
- **Status:** Manual deployments; not part of current scripted path

### Intended Architecture (From AgenticResolution_Architecture.md)

| Component | Service | Status |
|-----------|---------|--------|
| Python Resolution API | Container App (external ingress) | ✅ DEPLOYED |
| MCP Server | Container App (internal ingress) | ❌ NOT DEPLOYED / Phase 2 Candidate |
| Foundry Agents (triage + summarizer) | Azure AI Foundry | 🔄 Phase 2 (not Container Apps) |

---

## Decision Framework: Three Options

### Option 1: Declare Current Topology as Correct ⭐ RECOMMENDED

**Topology:** 2 container apps (API + Resolution)

**Rationale:**
- Phase 1 requirements complete
- Foundry agents (Phase 2) live in Foundry, not Container Apps
- MCP Server design completed but deprioritized; Phase 2+ scope
- Simpler architecture = lower cost for PoC

**Actions:**
1. Delete stub modules (`containerapp-mcp.bicep`, `containerapp-agent.bicep`)
2. Update Architecture doc to remove MCP Server from Phase 1
3. Lock this topology as Phase 1 final

**Owner:** Architecture decision

### Option 2: Deploy MCP Server to All Environments

**Topology:** 3 container apps (API + Resolution + MCP)

**Rationale:**
- Aligns with Architecture doc design
- MCP Server centralizes tool invocation for Phase 2 agents

**Actions:**
1. Flesh out `containerapp-mcp.bicep`
2. Wire into setup script
3. Deploy to both dev and test

**Owner:** Hicks (Backend)  
**Effort:** 2–3 days

### Option 3: Environment-Specific Topology

**Topology:** Dev gets MCP Server; test stays at 2 containers

**Rationale:** Dev for experimentation; test for cost control

**Trade-off:** Operational complexity; dev/test parity breaks

**Not Recommended:** Increases maintenance burden during Phase 2

---

## Recommendation

**Vote for Option 1** because:
1. Phase 1 scope is complete
2. Foundry agents are not Container Apps
3. Simpler = lower demo cost
4. Setup script idempotency is preserved
