## Session 2: Resolution Pipeline Redesign — Question-Driven KB Retrieval
**Date:** 2026-05-04  
**Duration:** ~1 day  
**Status:** ✅ Complete + Committed to main

### Problem Addressed
Old pipeline performed "dumb" KB search using ticket short_description, picked top-1 result. Evaluator received raw KB article. Fundamentally limited auto-resolve accuracy (~40%).

**Key failure case:** INC0010091 "VPN split tunneling misconfigured causing slow cloud app access" — system failed to match it to "VPN Not Connecting" KB article because search term didn't capture specific problem.

### Solution: Enriched Agent with Iterative Search
Single DecomposerAgent with native MCP tool loop:
1. Understand core problem
2. Decompose into 2-4 specific answerable questions
3. Execute targeted KB searches (multiple calls per question)
4. Synthesize answers into structured ResolutionAnalysis
5. Pass to EvaluatorAgent (receives structured analysis, not raw KB)

### New Message Types
- `ResolutionQuestion` — structured question + KB sources + synthesized answer
- `ResolutionAnalysis` — core_problem + questions[] + preliminary_confidence
- Removed: `KBSearchResult` (breaking change; KB retrieval now internal to DecomposerAgent)

### Implementation Changes
**Files modified:**
- `src/agents_py/shared/messages.py` — added ResolutionQuestion, ResolutionAnalysis; removed KBSearchResult
- `src/agents_py/agents/decomposer/__init__.py` (NEW) — DecomposerAgent with full prompt + MCP tool use
- `src/agents_py/agents/evaluator/__init__.py` — updated prompt to receive ResolutionAnalysis
- `src/agents_py/agents/incident/__init__.py` — simplified to pure data fetcher (emits TicketDetails)
- `src/agents_py/agents/request/__init__.py` — simplified to pure data fetcher (emits TicketDetails)
- `src/agents_py/workflow/__init__.py` — full rewrite (Classifier → Incident/Request → **Decomposer** → Evaluator → Gate → Resolution/Escalation)
- `src/agents_py/devui_serve.py` — registered decomposer_agent

### Workflow Architecture
```
Classifier (gpt-41-mini)
  ↓
Incident/Request Agent (data fetcher)
  ↓ TicketDetails
DecomposerAgent (NEW — key innovation)
  ├─ Understand problem
  ├─ Generate 2-4 questions
  ├─ Search KB multiple times (targeted queries)
  └─ Synthesize answers → ResolutionAnalysis
  ↓
EvaluatorAgent (receives structured analysis)
  ├─ Review questions + answers
  ├─ Determine if collectively resolve ticket
  └─ Assign confidence (0.0–1.0)
  ↓
Confidence gate (≥0.80)
  ├─ YES → ResolutionAgent (auto-resolve)
  └─ NO → EscalationAgent (escalate)
```

### Expected Impact
- **Auto-resolve rate:** ↑ from ~40% to 60%+
- **Latency:** +5–10 seconds (multiple KB searches)
- **Cost:** +$0.0003/ticket (acceptable for demo)
- **Transparency:** Questions + answers logged for failure diagnosis

### Tests/Validation
- All Python agents tested locally against devui
- Message types validated (dataclass serialization)
- Workflow graph wired and verified in devui execution traces

## Session 3: Hosted Agent Containers Migration
**Date:** 2026-05-04  
**Duration:** ~1 day  
**Status:** ✅ Implemented

### Problem
Deprecated `Azure.AI.Agents.Persistent` SDK with broken `require_approval` feature on East US 2 — agent runs hang indefinitely waiting for approval confirmations.

### Solution
Migrate to **3 containerized ASP.NET Core 9 hosted agents** inside Azure Container Apps:
- `classifier-agent` (CA) — classifies incident vs. request
- `incident-agent` (CA) — handles incidents (searches resolved tickets)
- `request-agent` (CA) — handles requests (identifies standard/complex types)

Each agent:
- **No Foundry SDK dependency** — calls MCP server directly via HTTP (JSON-RPC 2.0)
- **Invocations protocol:** `POST /invocations` accepts `{ Message, SessionId }`
- **Health endpoint:** `GET /health` for Container Apps liveness probes
- **Containerized:** Dockerfile for CA deployment

### Changes Made
1. **Created 3 new ASP.NET Core 9 projects:**
   - `src/Agents.Classifier/` — system prompt, MCP client, returns classification JSON
   - `src/Agents.Incident/` — system prompt, incident resolution logic, returns action JSON
   - `src/Agents.Request/` — system prompt, request type identification, returns action JSON

2. **Simplified Foundry wiring:**
   - Removed SDK polling loop for `RequiresAction` status
   - Direct HTTP POST to each agent service
   - No approval gate blocking

3. **Orchestration in API:**
   - New `AgentOrchestrationService` (calls classifier → routes to incident/request → combines results)
   - Wired in `WebhookDispatchService` for ticket processing

4. **Bicep infrastructure updates:**
   - Added 3 environment variables to API Container App: `Agents__ClassifierUrl`, `Agents__IncidentUrl`, `Agents__RequestUrl`
   - Post-deployment: populate URLs after agent containers deploy

### Trade-offs
**Pros:**
- No approval gate blocking
- Independent scaling per agent
- Direct MCP calls (simpler, more observable)
- Resilient to SDK changes

**Cons:**
- 3 additional Container Apps to manage
- Requires Bicep IaC for new services
- Agent URLs must be wired post-deployment
